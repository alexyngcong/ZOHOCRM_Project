Executive Summary

MICS-WF-V4-2026
PAGE 03 / 46

MICS International delivers comprehensive financial, tax, audit and business advisory services to clients across the UAE and GCC. This document codifies the operating system that powers every client engagement — from first touch to renewal — across a single Zoho-based technology stack.

System Purpose

The MICS Zoho ecosystem unifies Sales, Execution, Finance, HR, Marketing and Admin into a single source of truth. It replaces a previously fragmented process where departments used separate tools and shared logins obscured individual accountability.

◉ Core Operating Principle
Every record, every action, every notification carries a traceable Employee Code. Shared logins are permitted for cost reasons, but every submitted record captures which individual staff member performed the action, enabling full personal accountability within a shared infrastructure.

System Overview & Core Rules

MICS-WF-V4-2026
PAGE 04 / 46

Seven Unbreakable Rules

These rules are enforced by Zoho Flow automations and cannot be bypassed by any user role, including Partners. They ensure data integrity, accountability and client experience consistency.

Rule 01 · Employee Code Attribution
Every lead, contact, deal, invoice, work order, document and task records the Employee Code of the person who created or last modified it. Shared logins do not bypass this — the code is entered at record creation.
Rule 02 · Mandatory Payment Gate
No Work Order is created and no execution can begin until the agreed deposit (typically 50%) is received and reconciled in Zoho Books. The gate is enforced automatically.
Rule 03 · Ownership Is Set At Creation
Manually created leads, contacts and accounts are owned by the person who created them. Web/chat inquiries are assigned via round-robin OR to the first responder. Ownership transfers require a logged reason.
Rule 04 · Lead Source Owner Is Permanent
The person or source that brought a lead to MICS is permanently recorded and never changes — even if the lead owner changes. This preserves attribution for marketing ROI and referral tracking.
Rule 05 · Handler History Is Append-Only
Every owner change is recorded as a new Handler History entry with timestamp and reason. No entry can be deleted or edited — only new entries added.
Rule 06 · Execution Sees Everything
Once assigned, the execution team has FULL visibility into their client's data: signed contract, fees, portal credentials (FTA, DED, MOF, GDRFA, ICP), service history, BD owner and payment status.
Rule 07 · Every Action Is Audited
All record views, edits, exports, shares and password reveals are written to an immutable audit log, keyed by Employee Code, timestamp and record ID. Retained for 7 years.

Employee Code System

MICS-WF-V4-2026
PAGE 08 / 46

The Employee Code is the primary key identifying every individual staff member across every Zoho app. It is the mechanism that allows shared departmental logins while preserving per-individual accountability.

Code Format

◉ Format: EMP-NNN
Employee codes are 3-digit sequential numbers prefixed with EMP-. Examples: EMP-001 (Founder), EMP-002, EMP-003, EMP-004 (Partners), EMP-005 (Associate Partner), EMP-046 (most recent hire). Codes are never recycled — once assigned they persist even after termination, to preserve historical attribution.

Why Shared Logins?

MICS operates on a licensed Zoho model where per-user pricing is a material cost factor. Rather than purchase 46 individual Zoho seats, the organization uses 9 shared departmental logins that map to the mics.ae email architecture. The Employee Code field then attributes every action to a specific person within that shared session.

Ownership Transfer Flow

  1. Current owner initiates transfer by clicking Reassign on the lead record.
  2. Select new owner from Employee Code dropdown (scoped to BD + Managers).
  3. Provide reason (dropdown: Seniority / Specialty / Workload / Out of office / Other + free text).
  4. New owner receives notification via Cliq + email + CRM alert.
  5. New owner must accept within 24 hours; if not, escalates to Department Head.
  6. Handler History entry created automatically, permanent record.
◆ Transfer Is Not Deletion
When ownership transfers, the previous owner still appears in the Handler History. This is intentional — it prevents attribution gaming and ensures future sales analysis can distinguish between "brought the lead in" vs. "closed the deal."

  • Secondary Contact(s) — optional at this stage, can be added later
  • Accounting Contact (for Books integration — receives invoices)
  • Signing Authority (for engagement letters — may differ from primary)

Step 3 · Deal Creation

A Deal record is created representing the opportunity in play:

  • Deal Name (auto-suggested: "Account Name · Service · Year")

  • Expected Close Date

  • BD Owner = Lead Owner (pre-filled, can change)
  • Execution Assignment (optional at this point, set when Deal moves to Negotiation)

  2. Account record created with all enrichment data.
  3. Contact record(s) created, linked to Account.
  4. Deal record created, linked to Account + Contacts.

◉ Rule 1 · Creator Becomes Owner
Every manually created lead, contact or account inherits the creator's Employee Code as its owner. This is the default, and it prevents "orphan" records with no accountability.

Inbound inquiries via website form, chat widget, WhatsApp or hello@ email do NOT have a pre-existing owner. They are routed via two mechanisms: round-robin (default, distributes fairly) or first responder (whoever picks it up first owns it).

A Senior Tax Associate delivering VAT returns for 40 clients does not own any of them. The BD Owner / Account Manager retains the sales relationship. This allows execution specialists to focus on delivery quality without feeling obligated to upsell.

The Lead Source Owner never changes. It tracks WHERE the lead came from (a BD rep's outbound effort, the marketing website, an event, an external referrer). This preserves attribution accuracy for both revenue crediting and marketing ROI calculation.
◉ Rule 7 · External Referral Source Is Permanent
If the lead came via an external third party (partner firm, freelance referrer, employee of another company), that referrer is recorded permanently. This powers the external commission tracking (§12).

Security Controls

  • Encryption: AES-256 at rest, TLS 1.3 in transit
  • Access: Only the Execution Owner(s) assigned to that client's services can view credentials; other staff see masked values
  • Audit: Every password reveal is logged with Employee Code + timestamp + IP
  • Rotation: Credentials rotated quarterly for Banking, annually for government portals

  • Departure protocol: When an employee leaves, their view access is revoked within 1 hour; ability to log into shared departmental login also revoked
